# 简历写作 Skill

## 核心原则
写简历不是写博客。**高深、高级、极客、抽象**。

1. **不讲故事，不记流水账** —— 痛点、背景是面试时讲的，不是简历上写的。
2. **用精炼语言表达技术方案** —— 不堆实现细节，那会显得很 low。
3. **你是设计者，不是执行者** —— 描述方案是架构师视角，描述过程是码农视角。

## 禁止行为
- ❌ 写痛点/背景（“针对xxx的痛点”“由于xxx问题”）
- ❌ 写字段名/表结构（“shareUsers varchar(1000)”）
- ❌ 写实现细节（如“先更 DB 再刷新缓存”，可抽象为“双写策略”）
- ❌ 平铺流水账（三步做完写三步，像操作手册）
- ❌ 过度解释（把面试官要问的全写出来，堵死提问空间）

## 结构公式（总分结构，两段式）

### 项目描述（有多牛逼）
一句话说清：平台定位 + 你的职责 + 量化成果（可选）

### 核心工作（凸显技术深度）
每条工作点采用 **三句话** 结构：
1. **第一句**：做了什么（方案名称/技术目标）
2. **第二句**：具体技术方案（技术选型 + 架构组合）
3. **第三句**：保证了什么 / 做到了什么（效果或能力闭环）

每条工作点总长度 **2-3 句**，只保留技术骨架，给面试官留足提问空间。

## 词汇升级表（低级 → 高级）
| 低级/细节感 | 高级/工程感 |
|-------------|-------------|
| 先更 DB 再刷新缓存 | 双写策略，保证数据一致性 |
| 用分隔符拼接多个用户 | 多用户共享关系维护 |
| 写了个插件拦截请求 | 入口级权限校验 |
| 调用大模型分析代码 | 多智能体协作范式/Workflow |
| 把工具封装成 MCP | 异构能力协议归一化 |
| Prompt 工程 | 任务分发与上下文传递机制 |
| 定时任务备份数据 | 定时增量快照 + 控制面回调 |
| 项目描述完整还原需求 | 一句话给出平台定位与规模 |

## 自检清单
- [ ] 有没有写痛点/背景？（有 → 删掉，留给面试讲）
- [ ] 有没有字段名、表名、配置文件路径？（有 → 抽象掉）
- [ ] 有没有超过 3 句的核心工作条目？（有 → 压缩）
- [ ] 读起来像操作手册还是架构设计？（前者 → 重写）
- [ ] 面试官看完有没有想问的？（没有 → 你写太满了，留白不够）

## 修改示例

### Before（执行者视角）
> 参与共享/团队环境建设，实现 CPU 容器的一对多共享协作。基于数据库 shareUsers 字段（多用户以分隔符串联存储）维护共享关系，引入 Redis Set 作为高速读缓存加速共享者查询，写入时采用“先更 DB 再刷新缓存”的双写策略，并通过定时全量对账保证最终一致性；在网关层结合 APISIX 自定义校验插件，拦截非共享用户请求实现入口级权限校验，保障多用户协同访问的安全可控。

### After（设计者视角）
> **团队共享 CPU 环境多用户协同**：实现容器多用户共享与权限管控。Redis Set 做共享关系缓存加速，双写策略 + 定时对账保证数据一致性；APISIX 自定义插件实现网关层入口权限校验。
